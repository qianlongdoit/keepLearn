### 浏览器错误类型
+ js执行错误、资源加载错误
  ```js
  window.onerror = function(msg, url, line, col error) {}
  ```
+ promise错误
  ```js
  window.addEventListener('unhandledjection', e => {})
  ```

+ DOM事件错误
  ```js
  // 资源错误不会冒泡，所以得在capture阶段
  window.addEventListener('error', e => {}, true)
  ```

#### 为什么有这么写不同错误的捕获方式
因为这些错误在浏览器的发生阶段、执行上下文、错误传播机制完全不同。
1. 语法错误 SyntaxError
  发生在代码解析阶段，尚未允许，无法被try...catch捕获。比如 const a =;

2. 执行错误
  发生在执行阶段，比如 const a = 1; a = 2;
  可以被 try...catch捕获，也会冒泡到window.onerror

3. Promise 未捕获的异常
  因为这个是微任务，浏览器等一轮事件循环（执行一次宏任务，清空当前全部微任务）确认你没写catch才触发 ‘unhandlerejection’
4. DOM 错误（资源加载/运行时API）
   发生时机是浏览器层，并非js引擎，有单独的事件流。必须使用capture，因为资源错误不会冒泡

#### 为什么必须分开？
从上面可以看出，他们来自三套不同的系统
+ 语法错误：js 引擎、编译前、 window.onerror
+ 运行错误： js引擎、运行时、try...catch/onerror
+ Promise错误：事件循环、异步、onhandledrejection
+ DOM错误：浏览器、资源/事件、addEventListener

### 前端监控什么类型错误
+ js运行错误
+ Promise错误
+ DOM错误
+ 接口请求错误
+ 白屏 & 崩溃。
  
#### 为什么白屏、崩溃要单独一类
上面的4种理论上涵盖了浏览器的全部错误，但是这只是有报错的情况，这个是可以被捕获的。很多情况页面没有报错，但是却无法正常渲染的白屏是无法上报的。  
```javascript
// 没数据
const data = null;

if(data) {
    renderPage(data);
}

// 接口请求成功，但是数据异常

// 内存泄漏、死循环
```
所以白屏的本质是没有渲染有效的内容，监测白屏要从元素的内容监控入手，比如元素是否为空，DOM采样（是否都是空元素）、。
